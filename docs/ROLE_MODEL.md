# Role Model — root / company admin / owner / admin

## Зачем здесь несколько сущностей

В проекте есть два разных уровня управления, и они не должны смешиваться:

- организационный уровень: кто является главным администратором конкретной компании;
- CRM/RBAC уровень: какие permissions получает пользователь внутри интерфейса.

Из-за этого в системе одновременно существуют `company admin`, `owner` и `admin`. Это не дубли, а разные оси управления.

---

## Сущности

| Сущность | Где хранится | Что означает |
|---|---|---|
| `root` | WordPress user `ID = 1` | Глобальный системный суперпользователь над всеми компаниями |
| `company admin` | `crm_user_companies.is_company_admin = 1` | Главный администратор конкретной компании |
| `owner` | CRM-роль `crm_roles.code = owner` | Полный набор CRM-permissions внутри компании |
| `admin` | CRM-роль `crm_roles.code = admin` | Почти полный набор CRM-permissions, но без системного `audit.view` |
| обычный пользователь | membership + одна или несколько CRM-ролей | Оператор без полномочий company admin |

---

## Кто за что отвечает

### `root`

- существует над CRM-моделью;
- не является обычным CRM-пользователем;
- видит и меняет глобальную матрицу ролей и permissions;
- может работать cross-company только в root-only контурах;
- не должен появляться в обычных company-списках пользователей.

Подробно: `docs/ROOT_ACCOUNT.md`.

### `company admin`

- это не CRM-роль, а флаг членства в конкретной компании;
- определяет, кто может управлять ролями пользователей внутри company contour;
- должен быть только у главного администратора компании;
- первый пользователь компании, созданный root, автоматически получает этот статус.

`company admin` отвечает на вопрос: кто главный в компании.

### `owner`

- это CRM-роль с полным набором прав;
- используется как техническая материализация полного доступа внутри компании;
- для `company admin` роль `owner` должна существовать всегда;
- при обычном редактировании ролей у `company admin` `owner` сохраняется автоматически.

`owner` отвечает на вопрос: какие максимальные CRM-permissions получает пользователь.

### `admin`

- это расширенная CRM-роль для сильного оператора или заместителя;
- даёт почти полный набор прав, но не делает пользователя `company admin`;
- сам по себе `admin` не даёт права менять матрицу permissions ролей;
- сам по себе `admin` не даёт неснимаемый контроль над компанией.

---

## Главный принцип

- `company admin` = кто может назначать роли пользователям своей компании;
- `owner` = полный CRM-доступ внутри компании;
- `admin` = высокий операционный доступ, но не главный административный статус;
- `root` = единственный уровень, который меняет саму RBAC-матрицу.

---

## Что можно делать

### Назначать пользователям уже существующие CRM-роли

Может:

- `root`;
- `company admin` своей компании.

Не может:

- обычный `admin`, если у него нет `is_company_admin = 1`;
- обычный пользователь.

### Менять состав permissions внутри роли

Может:

- только `root`.

Не может:

- `company admin`;
- `owner` без root;
- `admin`;
- обычный пользователь.

---

## Инварианты

1. `root` всегда выше любой CRM-роли и выше любого company membership.
2. `company admin` и `owner` не конфликтуют: первый описывает статус в компании, второй описывает CRM-доступ.
3. `company admin` должен сохранять `owner` автоматически, чтобы не потерять управленческий контур случайным редактированием ролей.
4. `admin` можно назначать любым пользователям как рабочую роль без передачи статуса главного администратора компании.
5. Матрица ролей и permissions редактируется только в root-only интерфейсе, а не на обычной company-странице `Users`.

---

## Практический сценарий

Нормальная иерархия выглядит так:

- `root` создаёт компанию и первого пользователя;
- первый пользователь компании автоматически становится `company admin` и получает `owner`;
- `company admin` может назначать другим пользователям роли `admin`, `manager`, `support` и любые другие существующие роли;
- если нужно дать сотруднику широкий доступ без передачи полного контроля над компанией, ему дают `admin`, но не `company admin`;
- если нужно поменять, какие именно permissions входят в роль `admin`, это делает только `root`.
