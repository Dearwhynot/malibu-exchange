# Фиксация договорённостей по страницам продукта

Дата фиксации: 2026-05-18

Важно:
Это не новый roadmap и не пересборка исходной идеи.
Это точная фиксация того, что было проговорено перед реализацией.

## Что нужно добавить в проект

- В проект нужно добавить 2 страницы:
  - `ROADMAP`
  - `PATCH NOTE` / более подходящее название на усмотрение реализации

## Зачем это нужно

- Чтобы можно было спокойно оповещать команду о планах на разработку.
- Чтобы можно было публиковать изменения по проекту.
- Чтобы в изменениях желательно были обязательные скриншоты функциональных узлов, которые подверглись изменениям.
- Чтобы добавить проекту ощущение серьёзности и оформленного продукта.

## Что было отдельно отмечено

- Возник вопрос, есть ли этим страницам место в navbar под аккаунтами всех компаний.
- Допустимым и желательным решением признано размещение этих страниц в общей навигации для всех компаний.
- Отдельно зафиксировано желание менять версию проекта в футере.

## Как это было интерпретировано перед реализацией

- `ROADMAP` сохраняется как `Roadmap`.
- `PATCH NOTE` переименовывается в `Release Notes`.
- Для проекта это звучит взрослее и лучше сочетается с версией в футере.

## Решение по размещению

- Страницы добавляются в sidebar для аккаунтов всех компаний.
- Они оформляются как отдельный продуктовый блок навигации, а не как часть операционных разделов компании.

## Решение по версии

- В футере отображается версия проекта.
- Версия должна меняться штатно, не через правку шаблона руками.
- Версия должна логически вести на `Release Notes`.

## Разделение смысла страниц

- `Roadmap` = планы, направления, этапы, статусы будущей разработки.
- `Release Notes` = только уже выпущенные изменения.
- Для `Release Notes` правило про скриншоты фиксируется как обязательное для заметных функциональных изменений.
- Для `Roadmap` скриншоты не обязательны по умолчанию.

## Источник UI

- Для `Roadmap` допустимо опираться на визуальные паттерны `timeline`.
- Для `Release Notes` допустимо опираться на карточки и gallery-подачу из шаблона.
- Используется существующий template source проекта, без изобретения нового дизайн-языка.

## MVP-рамка реализации

- Две новые backoffice-страницы.
- Автосоздание страниц через миграции проекта.
- Пункты в sidebar.
- Версия в футере.
- Root-only точка, где можно менять версию и быстро переходить к редактированию контента страниц.
- Без отдельного тяжёлого CRUD на первом этапе.
