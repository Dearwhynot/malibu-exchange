# Root Account — Архитектура и правила

## Что такое root

WordPress user `ID = 1` — это **мастер-аккаунт** системы.

Root существует **над** CRM-моделью, а не внутри неё. Он не является оператором, не принадлежит ни одной компании и не виден ни в одном списке пользователей CRM.

---

## Правило company_id = 0

Root — единственный пользователь в системе, у которого `company_id = 0`.

| Пользователь | company_id | Смысл |
|---|---|---|
| uid = 1 (root) | **0** | Системный уровень, выше компаний |
| Любой оператор | ≥ 1 | Привязан к конкретной компании |
| Оператор без компании | 0 | Ошибка конфигурации — доступ заблокирован |

В БД:
- `crm_user_companies` — у root нет строк.
- `crm_user_accounts.default_company_id` у root = `0`.

---

## API

### `crm_is_root(int $uid): bool`
Каноническая проверка на root. Всегда используй её — никогда не хардкодь `=== 1` в новом коде.

### `crm_get_current_user_company_id(int $user_id = 0): int`
- root → возвращает `0`
- оператор с компанией → возвращает `company_id`
- оператор без компании → возвращает `0` (заблокирован gate'ом)

**Важно:** код, использующий это значение как `org_id`, обязан явно обрабатывать случай root через `crm_is_root()` до проверки `=== 0`.

### `crm_user_has_company_or_root(int $user_id = 0): bool`
- root → `true` (проходит все gate-проверки)
- оператор с компанией → `true`
- оператор без компании → `false` (блокируется)

---

## Gate: блокировка операторов без компании

Хук `template_redirect` (priority 15) в `inc/companies.php` перехватывает каждый запрос:

```
Запрос → is_user_logged_in? → is_root? → has_company? → разрешить
                                ↑ да         ↑ да
                             пропустить   пропустить
                                              ↓ нет
                                         wp_die(403)
```

Root всегда проходит. Оператор без компании получает 403 с сообщением «Ваш аккаунт ещё не привязан к компании. Обратитесь к администратору.»

## Настройки (settings) и root

Root имеет `org_id = 0` — системный контекст, **полностью изолированный** от всех компаний.  
Company admin работает с `org_id = company_id`.

```php
// Паттерн в page-settings.php и inc/ajax/settings.php:
$org_id = crm_is_root( $current_uid ) ? 0 : crm_get_current_user_company_id( $current_uid );
```

В таблице `crm_settings` строки `(org_id=0, key)` и `(org_id=1, key)` — независимые записи.  
Fintech-шлюз, cron и callback-обработчики читают настройки по `company_id` конкретной компании и не трогают `org_id = 0`.

---

## Видимость root в UI

Root полностью невидим на уровне приложения:

- **`WP_User_Query`** — всегда исключать `uid=1` явно.
- **AJAX-ответы** — никогда не включать root в ответы, описывающие пользователей.
- **UI** — root не редактируется, не блокируется, не удаляется через CRM.

Canonical helpers для этого:
- `crm_is_root(int $uid): bool` — в `inc/rbac.php`
- `_me_ajax_deny_root(int $user_id): void` — в `inc/ajax/users.php`

---

## Правила для разработчиков

1. `crm_is_root()` — единственный способ проверить root. Хардкод `=== 1` запрещён.
2. Новый код, работающий с `crm_get_current_user_company_id()`, должен перед проверкой `=== 0` сначала проверить `crm_is_root()`.
3. Root не добавляется в новые таблицы, списки или UI-элементы.
4. Документация о root изменяется только в этом файле и `AGENTS.md`.
